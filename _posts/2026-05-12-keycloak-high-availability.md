지난 글에서 비밀번호 정책을 다뤘다면, 이번 글에서는 한 발 더 운영 쪽으로 들어가 봅니다. **Keycloak이 죽으면 어떻게 되는가** — 단일 장애 지점(SPOF)을 어떻게 풀어냈는지 정리해 보려고 합니다.

## 배경 — 왜 인증 서버의 가용성이 문제였는가

Keycloak을 운용하면서 가장 마음에 걸렸던 부분은 **SPOF 문제**였습니다. **다수의 사내 시스템**이 Keycloak에 인증을 위임하고 있어서, Keycloak 한 대가 죽으면 그 시스템들이 동시에 로그인 불가가 됩니다. **수천 명 단위의 사용자**가 한 번에 영향을 받을 수 있는 구조였고, 가용성 확보는 우리 팀의 우선 과제가 될 수밖에 없었습니다.

## Keycloak 클러스터링 방안 검토

### 캐시 동기화의 필요성

노드를 여러 대 띄우는 것만으로는 끝나지 않습니다. **Keycloak이 내부적으로 캐싱을 적극 사용**하기 때문입니다. 공식 문서 기준으로 Keycloak의 캐시는 두 종류로 나뉩니다.

- **로컬 캐시** : 각 노드 내부에만 보관, 동기화 X
  `realms`, `users`, `authorization`, `keys`
- **분산 캐시** : 클러스터 전체에 복제·동기화
  `authenticationSessions`, `sessions`, `clientSessions`, `offlineSessions`, `loginFailures`, `actionTokens`, `work`(로컬 캐시 무효화 전파용)

세션·로그인 실패 같은 휘발성 데이터가 분산 캐시에 들어간다는 점이 핵심입니다. 동기화가 깨지면 다음과 같은 일이 벌어집니다.

| 동기화 실패 | 결과 |
| --- | --- |
| `authenticationSessions` | 로그인 도중 다른 노드로 라우팅되면 인증 상태 사라짐 |
| `sessions` / `clientSessions` | 로그아웃해도 다른 노드에서는 세션 유효 |
| `loginFailures` | 노드별 카운트 → 브루트포스 방어 무력화 |
| `work` | 권한 변경이 일부 노드만 반영, 데이터 불일치 |

결국 클러스터링의 본질은 "노드 수"가 아니라 "캐시 동기화"에 가까웠습니다.

### 왜 노드를 3대로 잡았는가

Keycloak이 캐시 동기화에 쓰는 라이브러리가 **Infinispan**입니다. Infinispan 자체에 대한 정리는 [별도 글](/posts/infinispan-basics/)로 두고, 여기서는 노드를 3대로 잡은 이유에만 집중하겠습니다.

핵심은 Infinispan 분산 캐시의 기본값 **`num_owners=2`** 입니다. 같은 데이터를 2개 노드에 복제해 두겠다는 의미인데, 우리 구성에 대입해 보면 노드 수가 자연스럽게 결정됩니다.

| 노드 수 | 결과 |
| --- | --- |
| 2대 | `num_owners=2` → 사실상 전체 복제. 1대 죽으면 owner=1로 떨어짐 |
| **3대** | `num_owners=2` 분산이 의미 있게 동작. 1대 죽어도 owner=2 유지 |
| 4대+ | 현재 트래픽 규모에서는 비용 대비 이득 작음 |

여기에 **롤링 배포** 시나리오가 한 가지 더 붙습니다. 한 대씩 내려서 업데이트하려면 배포 중인 한 대를 빼고도 owner=2가 유지돼야 하는데, 이 조건에서도 3대가 최소선이었습니다.

## 우리가 택한 구조 — 3노드 + L4 VIP

### 구성도

```
              [L4 VIP]
                 │
       ┌─────────┼─────────┐
       │         │         │
   [Keycloak] [Keycloak] [Keycloak]
    Node #1    Node #2    Node #3
       │         │         │
       └─── Infinispan ────┘
          (캐시 동기화)
```

### 동작 방식

**트래픽 분산 — 라운드 로빈**
L4 VIP는 인증 요청을 라운드 로빈으로 3대에 분배합니다. 세션 어피니티는 따로 걸지 않았는데, Infinispan이 세션 데이터를 2개 노드에 복제해 두기 때문에 어느 노드로 떨어져도 owner 노드에서 조회가 가능합니다.

**노드 장애 시 — 헬스체크 기반 라우팅 제외**
한 노드가 죽으면 L4 VIP가 헬스체크에서 풀에서 자동 제외하고, 복구되면 다시 편입시킵니다. 사용자는 다운된 노드가 있었는지조차 모릅니다.

**세션 유지 — Infinispan 동기화에 의존**
다음 요청이 다른 노드로 라우팅돼도 세션 데이터가 다른 owner 노드에 복제되어 있어, 사용자는 로그아웃되지 않습니다.

## 장단점

### 장점
- **SPOF 해소** : 한 노드가 죽어도 남은 2대로 서비스 지속, 사용자는 장애를 인지 못 함
- **무중단 롤링 배포** : 인증 서버 점검 공지 없이 배포 가능
- **트래픽 분산** : 단일 노드 시절 대비 동시 인증 요청 여유 확보
- **Embedded Infinispan** : 외부 캐시 클러스터 운영 부담 없음

### 단점
- **인프라 비용 증가** : 서버 3대 + L4 VIP
- **세션 어피니티 미적용에 따른 효율 손실** : 매 요청마다 owner 노드 조회 비용 발생
- **운영 복잡도 상승** : 노드별 로그 분산, 통합 로깅/모니터링 체계 필요
- **분산 시스템 고유 이슈** : Split-brain, rebalancing, TCPPING의 정적 IP 의존(노드 IP 변경 시 이미지 재빌드 필요) 등 운영 시야에 추가

## 정리

3노드 + L4 VIP + Embedded Infinispan 구성은 다수의 사내 시스템·수천 명 단위 사용자에게 영향을 줄 SPOF를 가장 단순한 형태로 제거하기 위한 선택이었습니다. `num_owners=2` 기본값과 롤링 배포 시나리오가 노드 수를 3대로 자연스럽게 정당화해 줬고, 그 위에 L4 VIP가 헬스체크·라운드 로빈으로 올라타면서 외부에서 보기엔 인증 서버가 한 덩어리로 동작하는 구조가 만들어졌습니다.

운영을 시작한 이후로는 다행히 크게 마주친 이슈는 없었습니다.
