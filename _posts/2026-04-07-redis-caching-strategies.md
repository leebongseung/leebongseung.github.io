## 뭘 캐시에 넣고, 뭘 안 넣는가

"데이터가 많으니까 다 Redis에 넣자"는 **잘못된 접근**이다. Redis 메모리는 비싸고(RAM = 비용), 캐시-DB 정합성 관리 비용도 데이터가 많을수록 늘어난다.

**캐시 적재 판단 기준:**

| 기준       | 캐시 O                   | 캐시 X                |
| -------- | ---------------------- | ------------------- |
| 읽기/쓰기 비율 | 읽기 >> 쓰기 (코드 테이블, 설정값) | 쓰기 빈번 (주문 상태, 재고)   |
| 데이터 크기   | 작고 자주 조회 (부서 코드 100건)  | 크고 개별적 (주문 상세 수만 건) |
| 정합성 요구   | Stale 허용 (카탈로그, 프로필)   | 실시간 정확해야 함 (잔액, 재고) |
| 재생성 비용   | DB 쿼리 비쌈 (JOIN 3개 이상)  | 단순 PK 조회            |

**실무 예시 — 주문 시스템:**

| 데이터 | 캐시 여부 | 이유 |
|--------|----------|------|
| 상품 카테고리/코드 테이블 | O (TTL 길게) | 거의 안 바뀜, 모든 페이지에서 조회 |
| 최근 주문 목록 (10건) | O (TTL 짧게) | 자주 조회, 목록 수준이라 크기 작음 |
| 주문 상세 (개별) | △ (조회 빈도에 따라) | 특정 주문만 반복 조회되면 가치 있음 |
| 주문 상태 (배송중/완료) | X | 실시간 변경, 정합성 중요 |
| 재고 수량 | X | 동시성 제어 필수, 캐시 Stale이 치명적 |

> **판단 원칙: "이걸 캐시에 넣으면 DB 쿼리를 얼마나 줄이는가?"**
> 줄이는 양이 적거나, Stale이 치명적이면 넣지 않는다.

## 4가지 캐싱 패턴

### Cache-Aside (Lazy Loading) — 가장 많이 사용

1. 애플리케이션이 데이터 요청 시 **캐시를 먼저 조회**
2. Cache Hit → 바로 반환
3. Cache Miss → **DB 조회 후 캐시에 저장**하고 반환

캐시 갱신 책임이 **애플리케이션 코드**에 있다.

| 장점 | 단점 |
|------|------|
| 요청된 데이터만 캐싱 → 메모리 효율적 | Cold Start 시 Cache Miss 지연 |
| 캐시 장애 시 DB fallback 가능 | Stale Data 발생 가능 (TTL 만료 전) |
| 구현 단순, 직관적 | miss마다 DB + 캐시 쓰기 레이턴시 |

**적합:** 읽기 빈도 높고 쓰기 빈도 낮은 데이터 (프로필, 설정값, 카탈로그)

### Write-Through

1. 쓰기 시 **캐시에 먼저 기록**
2. 캐시 계층이 **동기적으로 DB에도 기록**
3. 양쪽 쓰기 완료 후 응답

| 장점 | 단점 |
|------|------|
| 캐시-DB 데이터 일관성 보장 | 쓰기 레이턴시 증가 (양쪽 동기 쓰기) |
| 쓰기 시점에 캐시 적재 → Hit율 높음 | 읽히지 않는 데이터도 캐시 → 메모리 낭비 |

**적합:** 데이터 일관성이 매우 중요한 경우 (금융, 세션)

### Write-Behind (Write-Back)

1. **캐시에만 기록**하고 즉시 응답
2. 캐시 계층이 **비동기로 일정 시간 후 DB에 일괄 기록** (Batch)

| 장점 | 단점 |
|------|------|
| 쓰기 레이턴시 매우 낮음 | **데이터 유실 위험** (캐시 장애 시) |
| DB 쓰기 횟수 최적화 (배치) | 구현 복잡도 높음 (비동기 큐, 재시도) |

**적합:** 쓰기 빈도 매우 높고 유실 감수 가능 (조회수, 좋아요, 로그 집계)

### Read-Through

1. 애플리케이션이 **캐시에만 요청**
2. Cache Miss 시 **캐시 계층이 스스로 DB 조회** 후 적재

| 비교 | Cache-Aside | Read-Through |
|------|-------------|-------------|
| **로딩 주체** | 애플리케이션 코드 | 캐시 계층 (CacheLoader) |
| **코드 결합도** | 캐시 + DB 양쪽 인지 | 캐시만 인지 |
| **인프라 요구** | 특별한 지원 불필요 | Redisson 등 CacheLoader 필요 |
| **실무 빈도** | 가장 높음 | 프레임워크 의존, 상대적으로 낮음 |

## 3대 캐시 장애 패턴

### Cache Stampede (Thundering Herd)

**발생:** 인기 키의 TTL 동시 만료 → 다수 요청이 동시에 DB 폭격

| 해결 방법 | 원리 |
|-----------|------|
| **Mutex Lock** | 하나의 요청만 락 획득하여 DB 조회, 나머지는 대기 후 캐시 재조회 |
| **PER 알고리즘** | TTL 만료 전 확률적으로 미리 캐시 갱신. 잔여 시간 줄수록 갱신 확률 증가 |
| **Jitter TTL** | TTL에 랜덤 오프셋 추가하여 동시 만료 방지 |

```java
// Jitter TTL 예시
long baseTtl = 3600;
long jitter = ThreadLocalRandom.current().nextLong(0, 300);
redisTemplate.expire(key, baseTtl + jitter, TimeUnit.SECONDS);
```

### Cache Penetration

**발생:** **DB에도 존재하지 않는 데이터**를 반복 조회 → 매번 DB까지 도달

| 해결 방법 | 원리 |
|-----------|------|
| **Null 캐싱** | DB 결과 없을 때 "없음" 자체를 캐싱 (짧은 TTL: 30~300초) |
| **Bloom Filter** | 존재 여부를 O(1)로 판단. "없다" = 확실, "있다" = False Positive 가능 |

Redis에서 `RedisBloom` 모듈(`BF.ADD`, `BF.EXISTS`)로 네이티브 지원.

### Cache Avalanche

**발생:** **대량의 키가 동시 만료** 또는 캐시 노드 장애 → 모든 요청이 DB로 몰림

| 해결 방법 | 원리 |
|-----------|------|
| **TTL Jitter** | 모든 키에 랜덤 오프셋 추가 → 동시 만료 방지 |
| **계층형 캐시** | L1(로컬: Caffeine) + L2(Redis). Redis 장애 시 L1이 버팀 |
| **Circuit Breaker** | DB 요청 임계치 초과 시 회로 차단, fallback 반환 |

## Spring Cache 어노테이션 적용

| 어노테이션 | 동작 | 캐시 전략 |
|------------|------|----------|
| `@Cacheable` | 캐시 hit → 메서드 미실행. miss → 실행 후 캐시 저장 | **Cache-Aside** 패턴 |
| `@CachePut` | 메서드 **항상 실행** + 결과를 캐시 갱신 | **Write-Through** 성격 |
| `@CacheEvict` | 캐시에서 해당 키 **삭제** | Cache Invalidation |

**주요 포인트:**
- `@Cacheable(sync = true)` → 동일 키에 하나의 스레드만 DB 조회 (**Stampede 방어**)
- `@Cacheable(unless = "#result == null")` → Null 캐싱 방지/허용 제어 (**Penetration 방어**)
- Spring의 `@Cacheable`은 **TTL을 어노테이션 레벨에서 지원하지 않음** → `RedisCacheConfiguration`에서 캐시별 TTL 설정 필요
- `@Cacheable`과 `@CachePut`을 같은 메서드에 동시 사용 비권장 (동작 충돌)
