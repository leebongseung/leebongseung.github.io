## 배경

권한 체계를 설계하면서 가장 먼저 마주한 질문은 단순했습니다.
**"누구에게, 어떤 기준으로 권한을 줄 것인가."**

그 "누구"는 사용자 개인일 수도, 부서일 수도, 사용자가 가진 특정 속성(소속 거점·직책·내부/외부 여부 등)일 수도 있었습니다. 케이스가 다양했기 때문에 모델을 정하기 전에 **우리 조직에서 실제로 요구되는 권한 패턴이 무엇인지** 먼저 정리해야 했습니다.

## 현황 분석

여러 시스템(시스템1~N)에 흩어져 있던 권한 요구사항을 모아 보니, 큰 줄기는 네 가지로 정리됐습니다.

**1) 사내 보안지침 기반 공통 요구**

- 외부 협력사 계정 만료일 관리(최대 6개월)
- 1인 1계정 원칙, 관리자 메뉴 접근 분리
- 업무에 따른 권한 차등 부여
- 퇴사·부서이동 시 권한 회수/초기화
- 계정·권한 변경 로그 최소 5년 보관

**2) 조직 기반(부서·거점)**

- 특정 본부/거점 소속만 시스템 사용 허용 (예: 거점1·거점2 소속만 계정 생성)
- 조직 코드를 이용한 자동 권한 부여
- 겸직 시 최상위 직책 기준으로 단일 권한 부여

**3) 사용자 속성 기반**

- 내부/외부 사용자 구분에 따른 권한 차등
- HR 시스템에서 수신한 속성(직책·소속 등)을 인가 판단에 활용
- 외부 협력사 사용자는 별도 데이터로 관리

**4) 사용자 단위 직접 부여**

- 관리자가 개별 사용자에게 권한 직접 부여/회수
- 한 사용자에게 여러 권한이 동시에 부여될 때 **기본(primary) 권한** 지정
- 기간제 권한 부여(예: 외부 협력사 한시 권한)

## 결론 — 우리에게 필요한 모델

분석 결과, 우리 조직의 권한 요구사항은 결국 세 가지 축으로 환원됐습니다.

| 모델 | 풀네임 | 의미 | 비고(적용 단위) |
|---|---|---|---|
| **RBAC** | Role-Based Access Control | 사용자에게 부여된 **역할(Role)** 을 기준으로 권한을 결정하는 모델 | 부서, 직책, 시스템 사용자 단위 |
| **ABAC** | Attribute-Based Access Control | 사용자가 가진 **속성(Attribute)** 의 조합으로 권한을 결정하는 모델 | 소속 거점, 내부/외부 여부, 직책 속성 등 |
| **UBAC** | User-Based Access Control | 관리자가 **개별 사용자(User)** 에게 직접 권한을 부여/회수하는 모델 | 관리자 수동 부여, 기본(primary) 권한 지정, 기간제 권한 |

다른 모델(GBAC, TBAC 등)은 후보에서 제외하고, 위 세 가지를 **선제적인 설계 대상**으로 삼아 구현 계획에 들어갔습니다.

## 그렇다면 왜 권한 도메인을 직접 만들었는가

세 가지 모델(RBAC·ABAC·UBAC)을 정해 놓고 보니 다음 질문은 자연스럽게 따라왔습니다.
**"이 정도 요구사항이면 이미 누군가 만들어 둔 게 있을 텐데, 그대로 쓰면 안 되나?"**

### Keycloak이 풀어주지 못한 부분

먼저 짚어야 할 것은 **Keycloak이 무엇을 하고 무엇을 하지 않는가** 였습니다.

- 잘하는 것 — 인증, SSO, 사용자/역할(Role) 저장소, 토큰 발급
- 부족한 것 — **"어떤 규칙으로 권한을 줄 것인가"의 표현력**

우리가 정리한 요구사항을 다시 보면, 단순히 사용자에게 Role을 붙이는 수준이 아니었습니다.

- 조직 코드를 보고 자동으로 Role 부여
- 겸직 시 최상위 직책만 권한으로 인정
- 한 사용자에게 여러 권한이 있을 때 **기본(primary) 권한** 지정
- 외부 협력사 사용자는 만료일 기반으로 자동 비활성화
- 거점/내부외부 여부 등 **속성 조합으로 인가 결정**

Keycloak에서 Role 자체는 줄 수 있지만, **"어떤 조건에서 어떤 Role을 줄지"** 를 표현하기에는 한계가 분명했습니다.

### 오픈소스 대안 비교

그래서 후보를 몇 개 추려 비교했습니다. 비교 기준은 네 가지로 잡았습니다.

1. **정책 표현력** — RBAC/ABAC/UBAC를 모두 표현할 수 있는가
2. **관리자 UI** — 운영 담당자가 코드 수정 없이 권한을 부여/회수할 수 있는가
3. **조직·HR 동기화** — 조직도, 직책, 거점 같은 사내 데이터를 자연스럽게 받아올 수 있는가
4. **감사 로그** — 보안지침에서 요구하는 5년 보관 요건을 충족할 수 있는가

| 오픈소스 | 모델 | 강점 | 우리 케이스에서의 한계 |
|---|---|---|---|
| Casbin | RBAC / ABAC DSL | 정책을 정책 파일로 분리, 다국어 지원 | 조직 동기화·관리자 UI·승인 워크플로우는 직접 만들어야 함 |
| OPA (Open Policy Agent) | ABAC / Policy as Code | Rego로 거의 모든 정책 표현 가능 | 학습 곡선·운영 부하 큼, 관리자가 GUI로 권한 부여 불가 |
| Spring Security ACL | 도메인 객체 단위 | Spring 생태계 친화, 객체 단위 ACL | 사용자 속성·조직 기반 룰 표현력 약함 |
| Cerbos | 정책 서버 | 정책-앱 분리, 감사 로그 내장 | 겸직·primary role 같은 조직 특화 룰을 그대로 못 담음 |

공통적으로 드러난 결론은 명확했습니다.

> **정책 엔진은 잘 갖춰져 있지만, 그 위에 올릴 "조직 도메인"은 결국 우리가 만들어야 한다.**

### 직접 개발하기로 한 세 가지 이유

비교 결과를 추리면 자체 개발 결정의 근거는 세 가지로 압축됐습니다.

1. **조직 특화 룰을 그대로 표현할 수 있어야 했다.** — 겸직 시 최상위 직책, primary role, 거점별 데이터 분리, 외부 협력사 만료일 등은 어떤 오픈소스도 그대로 받아주지 않았습니다.
2. **운영 주체가 개발자가 아닌 관리자였다.** — Policy-as-Code로 가면 권한 변경마다 코드 PR이 필요해 운영 부하가 커집니다. 관리자가 GUI에서 직접 부여/회수하고, 그 이력이 그대로 감사 로그로 남는 구조가 필요했습니다.
3. **책임 분리가 명확해졌다.** — Keycloak은 **인증과 Role 저장소** 로만 두고, 그 위에 조직 규칙을 표현하는 **권한 도메인** 을 별도로 둔다. 이 분리가 장기 운영 관점에서 가장 깔끔했습니다.

이 결론을 바탕으로, 우리는 Keycloak 위에 자체 권한 도메인을 얹는 방향으로 설계에 들어갔습니다.
