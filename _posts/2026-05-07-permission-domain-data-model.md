지난 글에서는 우리가 왜 권한 도메인을 직접 만들기로 했는지를 정리했습니다. 이번 글에서는 **그래서 실제로 어떻게 만들었는가** — 도메인의 핵심 요구사항부터, 권한 부여 방식, Keycloak과의 동기화 전략까지를 다뤄 보려고 합니다.

## 권한 도메인의 핵심 요구사항

설계에 앞서 우리가 먼저 합의한 핵심 요구사항은 세 가지였습니다.

1. **사내 모든 관리 시스템의 권한을 한 곳에서 부여할 수 있을 것**
2. **모든 권한 부여/회수 이력은 빠짐없이 기록될 것** (보안지침의 5년 보관 요건과도 맞물림)
3. **특정 조건이 성립할 때 권한이 자동으로 부여되거나 회수될 것**

1번과 2번은 비교적 단순합니다. 모든 시스템의 권한을 단일 도메인에서 관리하고, 변경 이력을 audit 테이블에 남기면 됩니다. **진짜 문제는 3번** 이었습니다.

## 부여 방식 — 수동과 자동, 두 갈래

권한 부여는 결국 두 갈래로 나뉘었습니다.

- **수동 부여** — 관리자가 특정 사용자에게 직접 부여/회수 (UBAC 영역)
- **자동 부여** — 사용자의 속성·역할·재직 상태 등 조건이 충족되면 시스템이 자동으로 부여 (RBAC·ABAC 영역)

수동 부여는 단순합니다. 어려운 쪽은 자동 부여였습니다. **"어떤 조건에서 어떤 권한을 줄 것인가"** 를 데이터로 표현해야 했고, 그 조건이 단순한 단일 항목이 아니라 **여러 속성의 조합**이었기 때문입니다.

## 자동 부여의 핵심 — 집합 기반 조건식

이 문제를 풀기 위해 우리가 채택한 구조는 **집합(Set) 기반의 조건식 트리** 였습니다. 사용자에 대한 정보를 기준으로 AND/OR/괄호를 자유롭게 조합해 조건을 표현할 수 있게 한 것입니다.

```
(재직상태 = 재직중) AND (직급 = 팀장)
(권한 = ADMIN) OR (조직코드 = HQ001)
((재직상태 = 재직중) AND (소속거점 = 거점1)) OR (권한 = SYSTEM_ADMIN)
```

실제 운영에서 표현되는 룰을 예로 들면 다음과 같습니다.

- **"재직 중이면서 팀장 직급인 사용자"** → 결재자 권한 자동 부여
- **"ADMIN 권한을 가진 사용자"** → USER 권한도 함께 자동 부여
- **"외부 협력사 소속이면서 만료일이 지난 사용자"** → 모든 권한 자동 회수

이 구조의 장점은 분명했습니다.

- **세 가지 모델을 단일 구조로 수용 가능** — RBAC(역할 기반), ABAC(속성 기반), UBAC(사용자 기반)이 모두 같은 조건식 위에서 표현됨
- **관리자 GUI에서 룰 편집 가능** — 정책을 코드가 아닌 데이터로 관리하기 때문에, 운영 부서가 코드 수정 없이 룰을 추가·변경할 수 있음
- **확장 용이** — 새로운 사용자 속성이 생겨도 조건식에 항목 하나만 추가하면 됨

## Keycloak과의 동기화 전략

남은 문제는 하나였습니다. **"권한 도메인에서 결정한 권한을 Keycloak과 어떻게 맞출 것인가."**

Keycloak에는 권한과 관련해 두 가지 개념이 있습니다.

- **Role** — 사용자에게 부여되는 단순한 역할
- **Authorization** — 리소스·정책·스코프 기반의 세분화된 인가 모델

이론적으로는 Keycloak의 Authorization 탭에서도 RBAC·ABAC를 어느 정도 구성할 수 있습니다. 하지만 우리는 의도적으로 **Keycloak에는 Role이라는 단일 개념만 채택**하기로 했습니다. 이유는 두 가지였습니다.

1. **유지보수성** — 권한 표현이 권한 도메인과 Keycloak 양쪽에 흩어지면, 룰이 바뀔 때마다 두 곳을 모두 손봐야 합니다. 권한의 "결정"은 권한 도메인에서, "전달"은 Keycloak Role로 일원화하는 편이 훨씬 깔끔했습니다.
2. **Authorization 탭은 다른 용도로 살려둔다** — 추후 **메뉴와 권한을 엮는 작업**에서 Keycloak의 Authorization을 효과적으로 활용할 여지가 있었습니다. 미리 Role 외 다른 개념까지 끌어다 쓰면 그 자리를 잃게 됩니다.

결과적으로 동기화 흐름은 다음과 같이 정리됐습니다.

```
[권한 도메인] 조건식 평가 → 사용자별 부여 권한 결정
        ↓
[동기화 모듈] 변경분 추출 → Keycloak Role 부여/회수
        ↓
[Keycloak] Role을 토큰에 실어 각 시스템에 전달
```

## 결과

이 구조를 바탕으로 사내 다수의 내부 시스템이 동일한 권한 도메인을 통해 권한을 부여받을 수 있게 됐습니다. 운영 측면에서도 변화가 컸습니다.

- 관리자가 GUI에서 직접 룰을 편집·확인 가능
- 모든 부여/회수 이력이 audit 로그로 자동 누적
- 신규 시스템이 추가될 때 별도 권한 모델을 새로 설계할 필요 없음
